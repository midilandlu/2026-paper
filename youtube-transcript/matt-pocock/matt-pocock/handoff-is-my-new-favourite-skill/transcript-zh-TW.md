# /handoff 是我最新喜歡的技能

學習如何使用 /handoff 技能將您的 AI 寫程式對話移交給獨立的代理 (agents)。在這部深入探討的影片中，我將解釋為什麼我要建立這個技能、它與 /compact 有何不同，並展示管理多個並行 AI 對話的實際應用模式。

幾週前，我發現自己在和代理人 (agents) 互動時，做了一件自認為很聰明的事，但我以為那太簡單了，根本不需要寫成一個技能。對於不知道的人來說，我經常在思考技能這件事。我不斷思考如何將我的直覺和寫程式的習慣打包成可重複使用的技能，這也使得我的 skills 儲存庫在錄製這段影片時已經有將近 10 萬顆星星了。我開始思考的那個技能是一個交接 (handoff) [笑聲] 技能。其理念是，這個技能會把當前對話 (session) 的上下文視窗 (context window) 壓縮成一個 Markdown 檔案，然後可以將這個檔案移交給另一個對話。 [00:00:00 → 00:00:37]

因此，幾週前，我推出了這個功能。它位於 skills 資料夾內，在 productivity 裡，就是這裡的 handoff。這是一個非常、非常簡單的技能。它的說明是：寫一份交接文件來總結當前的對話，讓一個全新的代理人可以繼續進行工作。將其儲存到使用者作業系統的暫存目錄中，而不是當前的工作區。 [00:00:37 → 00:00:56]

我把這個放進我的 skills 資料夾做為實驗，想看看我會多頻繁地使用它。結果發現我用得非常多。在這支影片中，我將向你深入展示這個技能，例如我為什麼要設計它、它的重點是什麼、它與一些工具（像是 compact）中的內建功能相比如何，以及你該如何充分利用它，讓你的提問對話發揮最大效益。如果你喜歡我展示的這些內容，那麼你一定會喜歡我準備的課程，也就是給真正工程師的 AI 寫程式課程 (AI coding for real engineers)。這是一個為期兩週的梯次課程，專為想使用 AI 程式工具產出高品質程式碼，而不是生產垃圾程式碼的人所設計。 [00:00:56 → 00:01:31]

課程將於 6 月 1 日開始。我們現在有提供折扣。請點擊下方的連結查看詳情。首先，讓我們來解釋一下我為什麼要製作這個技能，以及它與你可能聽說過的「壓縮 (compaction)」有什麼不同。當我們在像這樣的寫程式對話中，基本上，隨著我們與代理人對話、當它呼叫工具 (tool calls)、當它編輯檔案時，這個上下文視窗就會被越來越多的東西填滿。 [00:01:31 → 00:01:57]

越來越多的 token 會填滿上下文視窗。現在，在我使用的工具 Claude code 中，上下文視窗非常大，對吧？你擁有 100 萬個 token 的上下文視窗，但這些上下文視窗實際上存在著「聰明區 (smart zone)」和「愚笨區 (dumb zone)」。在上下文視窗的早期，你會從代理人那裡得到更好的表現，因為那裡的注意力關係還沒有那麼緊繃。因為需要計算的 token 少得多，這些 token 之間的注意力關係也較少，所以代理人的注意力就不會那麼分散。 [00:01:57 → 00:02:31]

換句話說，當內容較少時，它能更好地集中注意力。這意味著隨著你們的對話發展，你從代理人那裡得到的回答會越來越笨，一直到你達到 80 萬個 token，不過我個人從沒遇到過，因為大約在 12 萬個 token 時，我就會開始覺得進入了「愚笨區」。所以，這意味著是的，儘管 Anthropic 為這些模型宣傳了大量的上下文視窗，但實際上，對於「真正需要聰明」的任務，你只有大約 12 萬個 token 可以運作，這表示你需要非常有效地規劃預算，並隨時注意你的上下文視窗。那麼問題就來了，當你開始碰上這個「愚笨區」時，你要怎麼做？你該如何恢復你的對話？ [00:02:31 → 00:03:16]

你該如何在保持聰明的同時，讓對話跨越「愚笨區」繼續進行？答案就是「壓縮 (compact)」。compact 的作用是，它會像這樣總結一個冗長的對話，所以你基本上可以從接近「愚笨區」一直回到這裡的「聰明區」。有時候甚至會有一個自動壓縮緩衝區，這取決於你使用的工具以及你是否開啟了該功能，這意味著當你接近上下文視窗的尾聲，比如說深入「愚笨區」時，自動壓縮緩衝區就會啟動，並在一個新的對話中自動總結你的對話。這個總結通常包含被參考的檔案（這只是一個被參考檔案的清單）、你在對話中說過的話通常也會被包含進去，以及對話的整體語氣。 [00:03:16 → 00:04:04]

然後這會作為一個小區塊包含在新對話的開頭，當你在新對話中建立上下文時，你會不斷地參考舊的對話。這意味著當你繼續不斷地壓縮時，你最終會因為先前的對話而形成這些不同層次的「沉積物」。這可能會有點效率低下，但如果你想進行某些特定類型的對話，也就是你需要一遍又一遍地處理同一個問題時，這也是個不錯的方法。這對於除錯 (debugging) 實際上非常有用，因為你可以把你嘗試過的所有其他選項都壓縮起來，然後繼續嘗試不同的方法，碰壁後再次壓縮，這基本上就是為了儲存你的狀態。所以，這是一種進行長時間對話的方式，但它實際上只是同一個對話。 [00:04:04 → 00:04:50]

所以，我一直覺得 compact 是建立這些冗長的單一對話非常有用的工具。但我開始注意到我想用 compact 做其他事情。我想壓縮到另一個對話中。例如，假設我在這裡的一個對話中，當我在這個對話中時，我注意到一個小小的重構機會。有些東西完全超出了我當前對話的範圍，但我知道我最終必須處理它。 [00:04:50 → 00:05:17]

那麼，我有什麼選擇呢？我可以延長我當前的對話，但這樣我最終會得到這種「被稀釋的上下文」，我一半在做一件事，一半在做另一件事，而且我絕對會碰到「愚笨區」，對吧？所以我可能無法完成我最初的目標。我可以執行壓縮，但那樣我就會把我當前對話中取得的所有進展都抹掉，對吧？我真正想做的是說：「好，我想在一個獨立的對話中完成另一件事，並保持我當前對話的純粹性。 [00:05:17 → 00:05:46]

」換句話說，這就是我想要的。我基本上想提取上下文，或者只提取與這個額外錯誤修復相關的部分，把它交接給另一個對話，然後這兩個對話就可以獨立運行。所以，有一段時間，我一直在做的是說：「好，把我當前對話中的東西拿來。我想修復這個特定的錯誤。幫我寫一份 handoff.md 文件，這樣我就可以把它傳遞給另一個代理人。」結果我發現我太頻繁地做這件事了，所以我決定：「好，我需要一個技能來做這件事。」 [00:05:46 → 00:06:05]

我最常在提問 (grilling) 時使用 handoff。在這裡，我正處於一個為了規劃 Sandcastle（這是我的一種軟體工廠）未來功能而進行的提問對話中。你可以看到我正在回答一些問題。 [00:06:05 → 00:06:29]

我才到這個提問對話的第二個問題 (Q2)，所以這不是一個很長的對話。我在這裡說：「我認為未來我們可能希望將迭代 (iterations) 和完成信號 (completion signal) 移到一個獨立的 API 上。事實上，我們把這個任務移交 (hand off) 給一個獨立的代理人吧。」你可以看到，當我定義 handoff 時，我正在說明移交的原因，以及那份文件中具體應該包含什麼。這會產生兩個作用。 [00:06:29 → 00:06:53]

首先，這實際上會讓我當前的提問對話更加聚焦。所以，它指出在該限制下，Q2 被折疊了。這實際上對我當前的提問對話有幫助，因為我在說明這超出了範圍，我們會在其他地方處理。接著它會創建一個 Markdown 檔案，內容包含下一個對話的重點、提交一個 GitHub issue，最終設計是將迭代和完成信號分離到獨立的 API 中。然後稍後，我只需將這個檔案傳遞給另一個代理人，讓它來建立 issue。 [00:06:51 → 00:07:22]

很簡單。另一個我非常強烈推薦的模式是，在提問對話中移交給代理人去建立原型 (prototype)。當你在進行提問時，當代理人透過 grill me 或 grill with docs（這些是我另外的技能）問你問題時，你通常會發現你需要回答的問題分為兩類。有一類是「已知的未知」，也就是代理人可以問你的問題，另一類則是需要真正在程式碼中看到或需要建立原型才能確定的東西。這對於像 UI 原型或你還不太確定如何處理的複雜邏輯片段來說尤其真實。 [00:07:22 → 00:07:54]

所以，在這個提問對話中，我們實際上進行到了第 13 個問題，並且我們從代理人那裡得到了一個類似最終的解決方案。然後你可以看到我說：「移交出去，去對這裡困難的部分建立原型，像是 window 通訊、teal draw SDK 整合」，這是我當時正在建構的東西。它建立了交接文件，然後我去那個分支上實作原型。所以在原型建立的對話中，這最終變成了一個龐大的對話，大概有 16.9 萬個 token，遠比能裝進提問對話中的還要大得多。我所做的是創建了這個 UI 原型和我希望看到的互動方式。 [00:07:54 → 00:08:29]

然後我說：「好，讓我們把它交接回衍生出這個任務的提問對話。從原型中提取所有的學習經驗，任何沒有直接在原型本身捕捉到的，或者那些非顯而易見的東西，給我一份交接文件，讓我能傳遞回給規劃者 (planner)。」這實際上是我在這裡非常常用的一種模式：你有一個初始對話用來做一些工作，然後你交接給另一個對話，該對話接著創建另一份交接文件，然後將它傳遞回給原始對話。這幾乎就像你做了一個 DIY 的子代理人，你可以使用一個上下文視窗來處理一個特定任務，壓縮你從那個任務中學到的東西，然後把它傳回給父任務。然後我就能完成提問對話，並利用原型建立適當的產品需求文件 (PRD) 和 issue。 [00:08:29 → 00:09:17]

所以，這是一個非常豐富的模式，可以從掛機的 (AFK) 代理人那裡獲得你需要的東西，並利用原型。非常非常酷。還值得一提的是，只在這裡使用像是 Markdown 文件而不是依賴某種原生的代理人功能的酷炫之處在於，你的第一個對話可以是 Claude code，但你可以直接把這個文件傳遞給另一個代理人，對吧？你可以把它傳給 Codex，或者傳給 Copilot CLI，無論你在使用什麼。所以，如果你想進行任何類型的對抗性審查，或者任何編碼代理人之間的互動，這是一個非常非常簡單的方法。 [00:09:17 → 00:09:51]

我們也應該通讀一下這個技能最後的部分，這樣你就能理解所有設計背後的邏輯。這裡的想法是，在文件中包含一個建議技能的部分，建議代理人應該呼叫的技能。我添加這個是因為，有時候我會使用技能來定義那個對話的風格，所以有一個建議技能的部分，意味著你只需將交接文件貼到新的對話中，它就會調用所需的技能，像是 grill with docs、diagnose 或 prototype 等等，然後你就可以準備開始了。所以，你不需要去思考你在下一個對話中需要使用哪些技能。 [00:09:51 → 00:10:27]

這非常方便。另一個是不要重複那些已經被捕獲在其他構件 (artifacts) 中的內容。我經常發現這些交接文件變得非常大，而且它們只是在重複已經存在於其他 Markdown 文件或像是 GitHub issue 等資源中的內容。所以，這基本上是在說，使用指標 (pointers) 就好，不要重複文件裡的所有東西。我也非常堅信，你應該將這些交接文件儲存到使用者作業系統的暫存目錄中。 [00:10:27 → 00:10:55]

換句話說，這些交接文件是用過即丟的。它們不是那種要被長期保留，然後在你的程式碼庫文件中腐爛的東西。另一個是遮蔽 (redact) 任何敏感資訊、API 金鑰、密碼或個人識別資訊 (PII)。這是非常必要的，你不會希望這些東西在隨機地方的 Markdown 文件裡飄來飄去。 [00:10:55 → 00:11:13]

最後，如果使用者傳遞了參數（也就是說下一個對話將用於什麼目的），請將其視為下一個對話將關注的重點描述，並據此調整文件。我認為這對交接來說至關重要，因為為了寫出一份像樣的文件，代理人需要知道下一個代理人對話的重點是什麼。每次我使用 handoff 時，我總是會描述移交的目的和原因，因為我想不出除此之外你要怎麼寫出一份好的交接文件。當然，語音輸入讓這一切變得很容易，因為我只要說出來，我們就可以繼續了。好了，這就是全部。 [00:11:13 → 00:11:46]
