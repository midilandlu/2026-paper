## [00:00:00] - 演講簡介與「技能地獄」概念

各位朋友好。我本來非常希望能親自參加 AI Engineer World's Fair，但因為家庭因素干擾，我無法成行。不過，我不會讓你們空手而歸。我會將原本要在舊金山發表的演講呈現給你們。這場演講叫做「失落的說明手冊：如何撰寫出色的技能 (The missing manual: How to write great skills)」。 [00:00:00 → 00:00:18]

我認為，區分好技能與壞技能的能力正變得越來越重要。作為開發人員，我們似乎很擅長為自己找到各種不同的「地獄」。像幾年前，我們有「教學地獄 (tutorial hell)」，也就是你會看一堆教學試著學習某些東西，卻無法將它們拼湊起來，然後陷入這種無法自拔的循環。我們還有「框架地獄 (framework hell)」，每十分鐘就有一個新的 JavaScript 框架發表，你必須不斷學習最熱門的新事物。 [00:00:18 → 00:00:49]

而現在，我認為我們有了另一種版本的地獄，那就是「技能地獄 (skill hell)」。技能地獄是指，你有所有這些可以免費取得的技能，你可以下載、貢獻、自己摸索，但你並不真正了解這些部分是如何協同工作的。你無法分辨技能的好壞。這表示人們正試圖將這些框架拼湊起來，試圖一次嘗試外面所有現成的東西。但他們做不到，或者說，他們沒有得到這些技能本身所承諾的效果。 [00:00:49 → 00:01:16]

這在個人層面是如此，在組織層面也是如此。組織沒有方法，也不了解如何建立好的技能。不知道如何將他們的標準作業程序 (SOP) 轉化為 Agent 可以執行的項目。如果你不這樣做，就很難獲得技能所能提供的豐碩成果。「兄弟，就再多加一個技能吧 (Just one more skill, bro)。」 [00:01:16 → 00:01:35]

這似乎就是我們現在的寫照。我也對此感到有些內疚，因為我們有 Matt POC Skills (我的技能 Repo)，這是目前最受歡迎的工程技能集之一。所以我覺得我想幫助使用我技能的人擺脫技能地獄。那麼我們要怎麼做？我們如何擺脫這種困境？ [00:01:35 → 00:01:52]

好吧，這裡究竟缺少了什麼？在我看來，我們缺少的是我們不知道「什麼造就了一個偉大的技能」。我們還無法看著一個技能說：「好吧，這個技能在這些方面做得很好，而在那些方面做得很差。」目前還沒有共通的準則 (rubric)，沒有一個可以評估技能並加以改進的框架。所以這就是我將在這次演講中提供給你們的。 [00:01:52 → 00:02:12]


## [00:02:12] - 技能檢核表框架概述

我將給你們一份技能檢核表 (skill checklist)。一份你可以用來檢查技能內部項目的清單，以確保它正在執行它所宣稱的功能，以及你可以改進它的方法、撰寫技能的方法。這份檢核表是這樣的：我們從技能的**觸發 (trigger)** 開始。技能是如何被呼叫的，以及你在這方面需要設計的決策。 [00:02:12 → 00:02:31]

接著是技能的**內部結構 (structure)**，技能實際上是如何組成與內部佈局的。然後第三點是，你實際上如何使用技能來進行**操縱 (steering)**？你如何讓技能告訴 Agent 該做什麼？然後第四點，你如何使技能盡可能地小？因為一旦我們有了一個有效的技能，我們接著需要基本上最大化它的效益。 [00:02:29 → 00:02:51]

**修剪 (Pruning)** 掉所有不相關的東西，修剪掉所有的「空操作 (no-ops)」。我沒有和你們在同一個房間裡有一個方便的好處，那就是你們可以立刻去嘗試這個，因為我已經將所有這些內容編碼到我的 Repo 中一個名為「撰寫優秀技能 (writing great skills)」的新技能裡。所以，如果你現在有迫切的需求，那就直接去我的 skills repo。你知道的，直接關掉這個瀏覽器，離開這裡，去使用這個技能來改進你現有的技能，或是寫出很棒的新技能。但讓我們先來過一遍這份檢核表。 [00:02:51 → 00:03:17]


## [00:03:16] - 觸發 (Trigger)：選擇使用者呼叫與模型呼叫的技能

首先是第一點，觸發 (trigger)，也就是技能被呼叫的方式。為了討論這個，我實際上要做一點比較：我的技能經常被拿來與另一套非常受歡迎的工程技能「Superpowers (超能力)」進行比較。我經常被問到一個問題：你的技能與 Superpowers 相比如何？它們之間有什麼差異？要理解這一點，我們需要了解**使用者呼叫 (user-invoked)** 與**模型呼叫 (model-invoked)** 技能之間的差異。 [00:03:17 → 00:03:40]

只要你有一個技能，你隨時都可以手動呼叫它。所以技能存放在你的檔案系統中。Agent 只要提取該技能並了解其中的內容即可。你可以透過與 Agent 溝通來隨時做到這一點。根據不同框架，不一定都是以「正斜線 (/)」的格式呈現，但你總是可以「由使用者呼叫」你的技能。 [00:03:40 → 00:03:59]

技能被呼叫的另一種方式是由 Agent 本身呼叫。這些被稱為模型可呼叫 (model-invocable) 技能或模型呼叫 (model-invoked) 技能。你可以設定一段描述。因此，技能的描述總是會進入 Agent 的 Context (上下文脈絡) 中，Agent 可以查看該描述並決定：「好的，根據這段描述，我將呼叫該技能，並最終讀取 skill.md 檔案」。 [00:03:58 → 00:04:20]

skill.md 就是技能核心所在，會被讀入我的 Context 視窗中。這就是呼叫技能的方式。這就是技能被呼叫時發生的事情。因此，這段描述就像是一種**上下文指標 (context pointer)**。它位於 Agent 的上下文中，指向另一個檔案，如果 Agent 需要更多關於該指標的上下文，它可以去那裡尋找。 [00:04:20 → 00:04:39]

你不需要把它放進 Agent 的上下文中。它對 Agent 可以是隱形的。而這就是我們所說的「使用者可呼叫 (user-invocable)」技能。因此，有些技能只能由使用者呼叫，因為它們沒有這個上下文指標。這是選擇性的。 [00:04:37 → 00:04:53]

例如，我們可以在我 Codebase 裡的設計看到，這是一個模型可呼叫技能。它有一個描述會進入 Agent 的 Context 視窗。但如果我們改看我的 `grill me` 技能，我們可以看到它有 `disable_model_invocation: true` 的設定。這表示這裡的簡短描述只會顯示給使用者看，Agent 是看不到的。 [00:04:53 → 00:05:11]

這就是技巧一：**決定你的技能是由使用者呼叫還是模型呼叫**。現在你可能會想，模型呼叫的技能更好，對吧？因為無論是模型自己呼叫，還是使用者呼叫都可以，這更有彈性。 [00:05:09 → 00:05:22]

但每次你在 Agent 的環境中新增一個模型呼叫技能，它就會增加我所謂的 Agent 的**上下文負載 (context load)**。它增加了一個新的描述，這在每次請求時都會消耗你的 Token，同時也增加了 Agent 需要思考的不同事物。所以如果你有 100 個模型呼叫技能，Agent 的上下文中就會有 100 條描述。所以看起來，要麼減少模型呼叫技能的數量，要麼全部使用使用者呼叫技能是比較合理的。但使用者呼叫技能有不同的負載，也就是使用者呼叫技能越多，對使用者造成的**認知負載 (cognitive load)** 就越高。換句話說，使用者需要記在腦子裡的事情越多，你對「駕駛員 (pilot)」的技能要求就越高。 [00:05:22 → 00:06:08]

因此，如果我們將 Matt Pocock 的技能與 Superpowers 進行比較，Superpowers 主要是模型呼叫技能。它賦予 Agent 超能力。而就我的技能而言，我更喜歡掌控全局。這表示我能盡量讓 Agent 的上下文負載保持在最低，但這確實對我施加了更多的認知負載。所以我需要非常深入地了解這些技能，才能充分利用它們。 [00:06:08 → 00:06:31]

那我為什麼要這樣做？為什麼我偏好使用者呼叫技能？因為，每次你有一個模型呼叫技能，你基本上都會付出「不可預測性」的代價，因為每次你有一個從一個資源指向另一個資源的上下文指標，模型可能就是選擇不跟隨它。你知道的，即使它絕對完美地適合該任務，它也可能選擇不呼叫該技能。我更喜歡消除那種不可預測性，轉而對使用者施加一點額外的認知負載。 [00:06:31 → 00:07:02]

你得到的好處就是消除了某種類型的問題，因為這種不可預測性讓人們需要去「評估 (eval)」他們的技能，以確保它們在正確的時間被呼叫，這是非常令人討厭的，而這是我寧可避免的問題。但我希望在這裡向你們展示的是，模型呼叫技能和使用者呼叫技能都有其各自的代價。所以選擇哪一個並不是個簡單的決定。這就是觸發 (trigger)，技能是如何被呼叫的。 [00:07:02 → 00:07:29]


## [00:07:29] - 結構 (Structure)：組織步驟與參考資料

現在讓我們來談談結構。技能的內部佈局。我認為在多數技能中你需要放入兩個主要單元。這兩個單元是**步驟 (steps)** 和**參考資料 (reference)**。步驟是技能將逐步走過的程序。 [00:07:29 → 00:07:46]

而參考資料是幫助它走過這些步驟的任何輔助資訊。你可以擁有沒有步驟、只有參考資料的技能。你也可以擁有沒有參考資料、只有一組簡單步驟要走的技能。但如果你開始將技能視為由這兩個單元組成，這真的有助於將它們進一步拆解。如果我們看一個例子，我的一個叫做 `two PRD` (轉為產品需求文件) 的技能，會從目前的 Context 視窗中建立一份產品需求文件 (PRD)。 [00:07:46 → 00:08:11]

它包含三個步驟。所以，它會找到相關的上下文。它會與使用者確認「測試接縫 (test seams)」。這裡有一個小型的人機協作 (human in the loop) 檢查點，只是為了確保我們在測試接縫方面沒有做任何奇怪的事情，我認為這非常重要。然後我們撰寫產品需求文件來處理這三個步驟。 [00:08:11 → 00:08:29]

我們有兩份參考資料。我們有一點關於「什麼是測試接縫」的參考。然後我們有一個產品需求文件的模板。就是一個實際的 Markdown 模板，用來撰寫 PRD。所以這是從頭開始撰寫技能的好方法。 [00:08:29 → 00:08:47]

你評估是否需要一些步驟。然後你寫下這些步驟，並評估這些步驟需要哪些參考資料，然後把它放在技能中一個獨立的小區塊，這就是給參考資料用的。然而，我們需要考慮一個非常重要的限制，這就是技巧三。 [00:08:45 → 00:09:03]


## [00:09:00] - 讓 skill.md 檔案保持精簡

我們希望讓主要的 `skill.md` 檔案越小越好。每個技能都是由它的描述，然後一個 `skill.md` 檔案，以及由此延伸的任何參考資料所組成。而這個 `skill.md` 檔案，如果我們讓它保持簡短，我們就能在很多方面節省成本。 [00:09:00 → 00:09:12]

較小的技能就是比較容易維護，容易審查，需要思考的詞彙更少。每次你刪減一個詞彙，那就是省下了一個 Token。這些累積起來的 Token 會從你的技能成本中扣除。所以我確實相信，不管是對維護者還是使用者來說，精巧的技能非常重要。 [00:09:12 → 00:09:35]

一個讓你的技能變小非常實用的方法是，思考技能的不同「分支 (branches)」，也就是技能被使用的不同方式。因為如果你有只在其中一個分支被使用的參考資料，那麼這就是從主 `skill.md` 中移除的候選項目。例如，如果我們看看這裡的 `two PRD`，我們有兩份參考資料。「什麼是測試接縫」和「PRD 模板」。 [00:09:35 → 00:09:58]

嗯，我們每一次都需要 PRD 模板，因為我們總是要建立一個 PRD。而且我們大概也每一次都需要「什麼是測試接縫」的資訊，因為我們總是要詢問測試接縫。所以對於 `two PRD`，只有一個分支，而且所有的參考資料都屬於這個分支。所以它大概也應該放在 `skill.md` 檔案裡。 [00:09:58 → 00:10:16]

然而，如果我們看我的另一個名為「領域塑模 (domain modeling)」的技能，它做了兩件事。它更新了一個名為 `context.md` 的本地詞彙表，然後它也會建立架構決策紀錄 (ADR)。換句話說，它正在做兩件不同的事情。或者它也可能選擇這兩件事都不做，在這種情況下，它不需要 `context.md` 模板，也不需要 ADR 模板。 [00:10:16 → 00:10:38]

所以換句話說，領域塑模有兩個或可能三個分支。這意味著我們不需要將 ADR 模板或 `context.md` 模板包含在主技能中。它們可以被移到獨立的區域。你的做法是使用 `skill.md` 檔案。 [00:10:38 → 00:10:53]

然後你把它放在一個上下文指標後面，並讓該指標指向技能資料夾內另一個獨立的 Markdown 檔案。這個上下文指標字面上只是說：「如果你需要模板，或者如果你需要更新 `context.md` 檔案，請去這個檔案」。我稱這為**外部參考 (external reference)**。 [00:10:53 → 00:11:11]

它是一個在 `skill.md` 外部的參考，你可以很容易地參照。Agent 可以很容易地把它拉進來，因為它與技能捆綁在一起。所以這是一個你可以用來讓 `skill.md` 盡可能小的技巧，這有非常多的好處。 [00:11:11 → 00:11:27]

**將分支參考資料隱藏在上下文指標後面**。換句話說，如果你覺得你的技能將以許多不同的方式被使用，那麼取出與這些分支相關的參考資料，將它們隱藏在上下文指標後面。這就是結構 (structure)。我們需要考慮讓 `skill.md` 變得超級小。 [00:11:27 → 00:11:44]

我們需要考慮技能中的分支，將素材移出到上下文指標後面。而我們需要考慮步驟和參考資料，這是技能裡面的兩個主要單元。 [00:11:44 → 00:11:55]


## [00:11:54] - 操縱 (Steering)：使用引導詞引導 Agent 行為

接著我們來談操縱 (steering)，也就是我們讓 Agent 做我們想做的事情的實際方法。對我來說，操縱歸結為一個非常酷的技巧，這幾乎是我希望你們從這次演講中學到的主要內容。這個技巧解決了一個問題，也就是「Agent 不做我想做的事」。換句話說，我在技能中指定了一些東西，我以為我已經說得很清楚了，然後它就是不做那件事。 [00:11:55 → 00:12:17]

我認為這種情況發生的主要原因是，你沒有使用一個叫做「**引導詞 (leading words)**」的技巧。引導詞的概念是，或者如果你喜歡文學理論的話，可以說是「主導動機 (leitmotif)」，那就是某些詞彙在非常小的空間裡包含了大量的意義。這些引導詞對 Agent 來說非常強大，因為你把引導詞放在技能本身的文本中，然後 Agent 會在它的操作過程中、在它的思考 Token 中、以及在它輸出給你的結果中，不斷向自己重複這個引導詞。然後，因為它重新強調了那個詞，而且那個詞希望就是描述了你對 Agent 的期望，這就會改變它的行為。讓我們用一個例子來讓它更具體。 [00:12:17 → 00:13:00]

## [00:16:48] - 修剪 (Pruning)：移除沉積物、無用資訊與空操作

現在我們進入修剪 (pruning)。修剪實際上就是一系列快速的失敗模式，也就是各種你可能會搞砸的地方。首先非常明顯的是，我們不想要龐大的技能。龐大的技能通常是另一種問題的症狀。這就是這些失敗模式的其中之一。 [00:16:49 → 00:17:09]

第一個非常簡單：**不要重複你自己 (Don't repeat yourself)**。你需要確保你留意重複的狀況。一般來說，我喜歡技能的每個部分都只有單一的真相來源 (single source of truth)。換句話說，如果你有一份參考資料，比如說 PRD 模板，或甚至是更小的東西，比如「什麼是測試接縫」，你要確保你不要在好幾個地方重複它，或者像是在多個地方涵蓋多個步驟。 [00:17:09 → 00:17:33]

只要確保每個部分都有單一真相來源，而且你也沒有在參考資料中重複自己。技能變大的下一個原因是「**沉積物 (sediment)**」。當人們在編寫同一套文件時，這真的是一個經典現象，也就是每個人都開始貢獻到一個共用的 Markdown 檔案。人們加上自己的東西。他們沒有勇氣去刪除和修改別人的內容。 [00:17:33 → 00:17:58]
