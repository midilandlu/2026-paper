所以我一直很想把它改名為 `to spec`，因為這才是我們真正在寫的東西——一份規格書。規格書是一個更廣泛的詞彙，更能精準涵蓋我們想構建的內容：一份關於我們想打造的事物的規格說明。這可以包含技術層面，也可以包含非技術層面，或者兩者兼具，這都無所謂。[00:01:23 → 00:01:36]

這件事真的讓我苦惱了很久，現在終於不再煩心了，反而覺得很開心。不過這個改名帶來了一個比較麻煩的地方：你可能需要刪除並重新下載這些技能。這代表你需要執行 `npx skills add mapco skills`。我非常確定這個技能安裝程式不會自動偵測到名稱變更，所以它不會主動幫你把 `to PRD` 更新成 `to spec`。[00:01:56 → 00:02:18]

這兩個技能都依賴一個核心的參考提問技能，它負責告訴 LLM 該如何向人類提問。我把這裡的敘述寫得更明確了，強調「一次問多個問題會讓人感到困惑」，即便原本就已經指示它「一次只能問一個問題」，它偶爾還是會一股腦地丟出好幾個問題。所以我現在明確告訴它「為什麼我們不希望它一次問多個問題」。我們也在最後面加上了一個確認機制。[00:02:41 → 00:03:09]

那就是「在我們確認雙方達成共識之前，不要執行計劃」。有很多使用不同模型的人回報說，提問階段有時會直接結束，然後就直接跳到實作階段。所以這算是多加了一道把關。最後，在某些情況下，它居然會「自己拷問自己」，這非常非常詭異。這不是我在自己的專案裡注意到的狀況，但我能看到的也只是這些技能實際使用情況的一小部分。[00:03:09 → 00:03:33]

所以我試著用一些引導詞，來區分「事實」和「決策」之間的不同。有時候，它是用之前的說法，也就是直接去探索程式碼並對自己提問。這在 fable 專案裡特別容易發生。所以我決定要把「事實」（也就是你透過探索程式庫自己發現的資訊）和「決策」分開來。[00:03:33 → 00:03:58]

而決策是需要由使用者來做出的。所以，我只是加了幾句話、調整了一些東西，這就讓它的表現一致多了。大家對那些奇怪問題的抱怨也確實少了很多。接下來要說的是，我新增了幾個技能，把原本主要偏向規劃、沒有太多實作引導的流程，轉變成一個完善的軟體開發生命週期。很多人問我，整個流程到底是什麼？[00:03:56 → 00:04:24]

使用這些技能時，主要的流程應該長怎樣？首先，這就是流程。第一步，你應該讓 AI 代理對你進行提問 (grill)，而不是使用 plan (計畫) 模式。它會利用這些文件建立詞彙表，在過程中更了解你，並且加入架構決策紀錄 (architectural decision records)，用來捕捉那些不那麼顯而易見的想法。透過查閱文件產生的提問內容，接著就會變成我們前面看到的那種規格書 (spec)。這份規格書定義了你開發的目標。[00:04:22 → 00:04:55]

然後，你把這份規格書拆解成一個個獨立的工單 (tickets)，這樣你就能把開發工作分散到多個不同的代理會話 (agent sessions) 中進行。這就是 `to tickets` 的作用。接著，你可以用 `implement` 技能來實作每一張工單。而這個 `implement` (實作) 技能其實非常簡單，它長這樣。[00:04:55 → 00:05:10]

「實作使用者在規格書或工單中描述的工作。可能的話，在事先同意的接縫處 (seams) 使用 TDD (測試驅動開發)。」這是個不錯的要求。「並定期執行型別檢查。定期執行單一測試檔案。」[00:05:10 → 00:05:22]

「最後在完成時，執行一次完整的測試套件。完成後，使用 `code review` (程式碼審查) 來檢查工作，然後將你的工作提交 (commit) 到目前的分支。」我本來差點不想做這個技能，因為它真的太簡單了，對吧？它主要只是仰賴代理本身的先驗知識，也就是藉由預設的指令來告訴它該怎麼做。老實說，我本來覺得這裡不需要一個專屬的技能。[00:05:20 → 00:05:39]

但是大家一直問我，流程是什麼？這裡的流程長怎樣？所以我想，乾脆就做一個 `implement` 技能，讓它簡單明瞭，對吧？在流程的每一個階段，就呼叫對應的技能。這代表 `implement` 贏得了它在這裡的地位，因為你知道的，一旦我們有了工單 (`to tickets`)，我們只需要在不同的寫程式會話中，實作 (`implement`) 每一張工單就可以了。[00:05:39 → 00:05:57]

接著 `implement` 本身會呼叫 `code review`。至於 `code review`，我把它從版本 1 的「進行中」狀態畢業了。所以它其實已經存在一段時間了，但我在這個版本 2 對它做了一些更新。`code review` 技能的理念是，它會從兩個層面來審查程式碼。所以它會為這兩個層面各指派一個子代理 (sub agent)。第一個是標準 (standards) 層面。[00:05:57 → 00:06:18]

「程式碼是否符合這個儲存庫中記錄的程式碼撰寫標準？」所以，如果你在儲存庫某處有一份 `coding-standards.md` 檔案，它就會去讀取並用它來檢查。我通常認為，程式碼標準不應該放在 `agents.md` 檔案裡面。[00:06:18 → 00:06:34]

它們應該放在獨立的地方，而在 `code review` 階段，才是它們最派得上用場的時候。完成標準檢查後，接著進入規格書檢查：「程式碼是否忠實地實作了最初的 issue、PRD 或規格書？」它們會以平行的子代理形式運作，並在這裡執行逐步檢查每一部分的流程。現在這個技能有個很酷的新功能，就是我又重新讀了 Martin Fowler 的《重構》(Refactoring)。[00:06:33 → 00:06:59]

我決定這麼做，是因為 Martin Fowler 在書裡列出了很多在糟糕程式碼中能被代理識別出的「壞味道」(smells)。《重構》是一本很經典、被廣泛引用的書，所以這些「壞味道」深深烙印在代理的先驗知識裡。所以你只需要喚起這個概念，例如「神秘的命名 (mysterious name)」、「重複的程式碼 (duplicated code)」、「依戀情結 (feature envy)」、「資料泥團 (data clumps)」、「基本型別偏執 (primitive obsession)」、「重複的 switch (repeated switches)」、「發散式修改 (divergent change)」、「夸夸其談未來性 (speculative generality)」、「過度耦合的訊息鏈 (message chains)」。你懂我意思嗎？這些都深植於代理的知識庫中，我們只需要用一句話來描述它們，我就發現這能引導代理把這個詞彙回饋給你，並告訴你：「沒錯，我發現了一些過度耦合的訊息鏈，我需要把它們移除；我發現了中間人 (middleman) 的情況，我需要去修正它。」所以我測試了幾週，發現它出乎意料地實用。它在提升程式碼品質方面表現得非常出色，而且加在這裡的成本很低，[00:06:59 → 00:07:51]

只需要大概 10 行程式碼。但是，我們來談談那個讓我超級超級興奮的功能，那就是啟動和塑造規格書的全新方式。也就是，前置規格 (pre-spec) 階段。換句話說，它被放在這裡，在某些情況下可能會取代 `Grill with Docs`，它叫做 `Wayfinder` (尋路者)。我會專門寫一篇文章、拍一支影片來介紹 `Wayfinder`，但簡單來說，我希望在某些你想使用 `Grill with Docs` 的情境下，可以改為預設使用 `Wayfinder`。[00:07:51 → 00:08:25]

`Wayfinder` 是為了解決你有一大堆東西想計畫，但規模太大，一個代理會話處理不來的情況。也就是說，這會超出代理的「智慧區間 (smart zone)」，甚至可能超出代理的上下文視窗 (context window)。你需要把它拆分成好幾個部分，才能弄清楚你的方向。「一個模糊的想法浮現了，規模對一次代理會話來說太大了，而且籠罩在迷霧之中。從這裡到目的地的路徑還看不清楚。」[00:08:25 → 00:08:51]

這個技能會在儲存庫的問題追蹤系統上，把路徑繪製成一張共享地圖，然後一次處理一張工單，直到路徑變得清晰為止。這些地圖會被儲存在 GitHub issues 中。舉例來說，這是 Sand Castle 儲存庫裡的一個例子，我們在做一個 Spike (探測性任務)，評估是否要把 AI SDK 納入依賴項。這是一個很大的變動。所以你可以看到，到目前為止還沒有做出任何決策，而所有需要做出的決策都被儲存為子 issues，這些子 issues 之間有阻擋 (blocking) 的關聯性。[00:08:51 → 00:09:21]

所以我們可以看到，在我們一開始做出這個關鍵決策之前，無法進行後續的任何決定。每一個決策的範圍，都被限定在一次代理會話可以處理的大小。我們還能看到它們被標上了不同類型的標籤。例如，這個被標記為研究 (research) 任務。所以，這真的是一個適合代理離線 (AFK) 去做研究，然後再回來回報的任務。[00:09:19 → 00:09:42]

最終你得到的結果是，在所有這些工單都關閉後，所有的資訊都會被儲存到地圖上，而原始的工單則作為捕捉這些資訊的主要來源。然後你就可以拿著這張地圖，用一般的方式把它轉變成一份規格書。我發現，現在我不需要再為了管理 `Grill with Docs` 的會話而感到焦慮，不需要擔心交接的問題、也不用擔心超出代理的「智慧區間」；有了 `Wayfinder`，這些都幫我管理好了，我只要關閉一個會話，打開下一張 `Wayfinder` 工單就可以了。這一切都儲存在 GitHub 上，所以是協作的。你可以與你的團隊分享。[00:10:45 → 00:11:19]

一旦地圖完成了，一旦它完整了，你只要去到 `to spec`，就可以準備開工了。為了支援 `Wayfinder` 技能，我們有一個新的 `research` 技能，它非常小，但非常方便，當你只需要做一個研究會話時，或者它可以在做研究時往正確的方向引導模型，至少是我喜歡的方向。「啟動一個背景代理來進行研究，這樣你就可以在它閱讀時繼續工作。對照主要來源調查這個問題。將發現寫成一個簡單的 markdown 檔案，並儲存在儲存庫中原本就存放這類筆記的地方，符合現有的慣例。」[00:11:19 → 00:11:47]

所以，它要麼是一個邏輯原型，要麼是一個 UI 原型，它們的反應截然不同。最後一項變動是大家要求了很久，而我終於決定扣下扳機實作的功能。之前，在我的 TDD 技能裡，它會推薦一系列步驟讓你遵循，但有時候這有點尷尬。那些步驟像是，它會跟你確認想寫哪些測試，然後你會帶著它一步一步做。但這並不符合多數人對 TDD 運作方式的想像。你應該要能丟一個 TDD 技能給離線 (AFK) 代理，然後它就應該自己搞定。所以這個 TDD 技能現在只作為參考資料，除了建議你撰寫測試的順序之外，不指定任何特定步驟。也就是說，要遵循「紅燈、綠燈、重構」的順序。它只是說，一次切一小塊，先紅燈再綠燈。它也把重構獨立出來，不再視為循環的一部分。所以它不再是一個「紅燈、綠燈、重構」的循環。[00:12:09 → 00:13:05]
