# AI 系統設計：從構想到投入生產 - Apoorva Joshi, MongoDB

寫程式不再是難事，AI 就能代勞。在 AI 撰寫程式碼的世界裡，工程師最有價值的技能就是知道該打造什麼。大多數 AI 系統之所以從未進入生產階段，都是因為在過程中過早做出了錯誤的決策。為了追求快速發布而承受的壓力，以及各種炒作...

大家好，我是 Apoorva，曾任資料科學家，目前在 MongoDB 擔任開發者倡導者（Developer Advocate）。職涯初期，我主要負責為各種網路安全案例打造機器學習應用程式；現在，我運用這些應用機器學習的知識，協助 AI 開發者使用 MongoDB 和 Voyage AI 成功打造 AI 應用程式。在這場演講中，你將學到如何從頭到尾（從構想到投入生產）思考並打造 AI 系統。我們將以一個真實世界的案例，帶你走過設計過程的所有步驟。希望在演講結束時，你能獲得一套可重複使用的框架，並將其應用於你設計的任何 AI 系統中。[00:00:02 → 00:00:44]

你可能會想，在 AI 時代，我們真的還需要思考該打造什麼嗎？只要直接寫程式碼（vibe code）然後發布不就好了嗎？但這正是我要切入的重點。當你只是為了好玩而開發、風險很低，且可以輕易用肉眼判斷輸出結果是否正確時，直接寫程式碼確實效果很好。[00:00:46 → 00:01:11]

但當你開發的是真實的產品、別人會依賴的東西、且有實質影響時，「直接發布」其實是有點危險的。而且不只有我這麼說，就連那些非常看好 AI 寫程式的 Anthropic 和 OpenAI 人員也是同樣的看法。這其實是他們過去幾個月演講中的引言：「規格（Specs）就是新的程式碼。」重點在於如何定義產品需求、系統設計和評估標準，這樣你才能確信你的 AI 程式設計助手正在打造正確的東西。[00:01:09 → 00:01:43]

接下來的演講都將圍繞著這個主題：我們該如何做好這件事？將其視為一個框架會很有幫助。這包含四個階段：首先是產品需求。[00:01:43 → 00:01:56]

你實際要打造什麼、為誰而打造，以及有哪些限制？接著是系統設計：能幫助你實際滿足這些需求的資料、架構和模式。再來是評估和監控：在發布之前與之後，你如何知道所打造的東西是否真的有效？最後，你該如何優化成本？在發布前或在生產環境中發現落差時，你不只要針對準確度，還要針對成本、延遲（latency）和可靠性進行優化。[00:01:56 → 00:02:29]

與其抽象地談論這個框架，我想不如將其應用於一個真實案例，這樣你們就能清楚看到每一個決策是如何從框架的一個階段推展到下一個階段。我們就以「健康保險理賠審查系統」為例。一直以來，健康保險理賠的審核都是一個極度依賴人工的流程；人類醫療審查員必須交叉比對臨床指南、保險理賠政策、病患病史等資訊，才能決定某項醫療、手術或藥物是否涵蓋在病患的健康保險保單範圍內。即便你住在有免費醫療體系的地方，你的醫療體系中仍會有些項目有給付、有些沒有。因此，你多半還是會經歷類似的流程。[00:02:30 → 00:03:29]

我們打造這個系統的目標，是看看能否借助 AI 來改善或簡化醫療審查員的使用體驗。我們來看看這個應用程式在框架中的「產品需求」階段會是什麼樣子。首先要做的是量化商業問題。商業問題必須針對特定事項，清楚說明這個應用程式的使用者是誰、目前的狀況，並量化使用者的痛點。它不應該規定系統會是什麼樣子，不管是代理人（agent）、多代理人系統還是其他形式。[00:03:27 → 00:04:11]

那會是之後才要決定的事。[00:04:08 → 00:04:11]

所以，我們的理賠審查系統面臨的商業問題可能是這樣的：MDB Health 的醫療審查員平均要花 2 天處理理賠審查請求，這是一般非緊急案件業界標準的 4 倍，更是緊急案件業界標準的 12 倍。這種規模的延遲會耽誤病患的照護，特別是那些有時間急迫性的治療。正如你所見，這個商業問題是針對特定使用者的；它明確指出這是為了醫療審查員而設計。[00:04:14 → 00:04:47]

它陳述了目前的狀況，也就是這種人工流程導致他們要花 2 天時間處理理賠審查。因為它提出了一些基準線，所以它是可衡量的。它不預設立場（solution-agnostic），並沒有真正告訴你系統應該長什麼樣子，而且它非常專注於一個特定的問題。接下來，你需要收集應用程式的任何商業限制。[00:04:44 → 00:05:17]

例如任何合規的監管需求、資料不能離開組織的限制，或是採購限制（意思是組織內是否禁用某些特定供應商）。在你開始設計系統之前，最好能了解所有的這些資訊。針對我們的應用程式，假設有以下這些商業限制：病患資料必須留在經過核准的雲端環境中；只能使用經過核准的雲端環境上的模型。[00:05:17 → 00:05:47]

有些案例是需要人工審查的。所以，任何複雜的案件都需要資深醫師的審核。還有什麼呢？任何拒絕理賠的決定都必須由人類審查員來審核。所以，有些情況是你無法完全用 AI 來自動化的。[00:05:47 → 00:06:06]

對，這些都是很有用的設計輸入資訊。所以你會希望越早取得這些資訊越好。[00:06:04 → 00:06:11]

此外，了解任何效能需求也是件好事。我們是否需要毫秒等級的回應時間，或者我們還有一些空間？我們每個月能在 LLM 推論上花多少錢？有任何需要注意的系統正常運行時間 SLA 嗎？其次，如果你正在開發一項 AI 產品，釐清 AI 在你的產品中所扮演的角色會很有幫助。[00:06:14 → 00:06:39]

我喜歡從三個面向來思考 AI 的角色。第一，AI 對你的產品來說是不可或缺的，還是輔助性的？在這個案例中，它是輔助性的，因為在沒有 AI 的情況下，理賠審查就已經在進行了，只是速度比較慢。其次，這會是一個被動（reactive）還是主動（proactive）的系統？意思是，它是由使用者或事件觸發的，還是會主動執行任務？[00:06:37 → 00:07:05]

在這個案例中，它將是一個被動的系統，因為它是藉由實際提交理賠申請來觸發的。最後，AI 在應用程式中的自主程度有多高？在我們的案例中，商業限制規定在某些情況下需要人工審查。因此，我們的系統最多只能是半自主的（semi-autonomous）。最後，作為定義解決方案的一部分，你也會想提出一到兩個與商業問題相符的成功指標，這樣你才能捕捉到成功的樣貌。[00:07:05 → 00:07:42]

一個好的成功指標是具體（specific）、可衡量（measurable）、可達成（achievable）、相關聯（relevant）且有時限的（time-bound）。這是一個針對我們理賠審查應用程式的成功指標範例。指標中寫道：在系統推出後的 90 天內，將緊急理賠審查請求的平均處理時間從 2 天縮短為 1 小時。它非常具體。因為它規定處理時間需要是 1 小時，所以它是可衡量的。[00:07:39 → 00:08:12]

這似乎是可達成的，因為會有 AI 介入其中，所以希望任何收集資訊或取得初步建議的步驟都會變得更快。這與先前提到的商業問題高度相關且根植於其中；而且它是有時限的，因為它給了我們一個必須開始看到成效的時間範圍。[00:08:09 → 00:08:32]

好的，到目前為止我們已經弄清楚了「要做什麼」。接著我們來看看「怎麼做」。這裡你要思考的第一件事是你的資料策略。你的應用程式需要什麼資料？你甚至有權限存取嗎？[00:08:35 → 00:08:52]

它存放在哪裡？它的原始格式是否已經準備好可供直接使用？諸如此類。針對我們的理賠審查應用程式，這些是我們可能需要的資料來源、它們存放的位置，以及存取原始資料的格式可能長什麼樣子。我們可能需要臨床指南，這描述了針對特定情況哪些護理是適當的。我們也需要 MDB Health 的理賠政策，其中說明了 MDB Health 將會與不會支付哪些費用。[00:08:52 → 00:09:24]

最後，病患的理賠紀錄，用以了解任何過去的病況、曾經進行過的手術、手術時間等資訊。假設臨床指南和理賠政策是以 PDF 格式儲存在像 Confluence 的地方，而理賠紀錄則是在處理過程中儲存於 MongoDB 內。接下來要思考的是資料更新頻率。資料來源多久更新一次？現在，如果你要對這些原始資料來源進行任何資料處理（而且你通常都會這麼做），這些資料管線（pipelines）就必須以正確的節奏運行以保持同步。[00:09:22 → 00:10:04]

否則，我們的系統就會根據過時的資訊運作，這並不好，尤其是在這種高風險的應用案例中。那我們就來為理賠審查系統整理一下。臨床指南可能每年更新一次；內部理賠政策可能是每季更新一次；而病患理賠紀錄必須在任何新理賠處理完畢時更新。在我們的成功指標中，如果你還記得，我們說過要將審核緊急理賠的目標時間設為 1 小時，所以這些資料可能需要每小時更新一次。[00:10:02 → 00:10:40]

其次，這些資料可以直接由我們的應用程式使用，還是需要任何處理？對於通常是長篇文件的臨床指南，你可能會想將它們分塊（chunk）、將分塊進行嵌入（embed），並擷取程序名稱、出版日期等詮釋資料（metadata），以使其適合使用向量搜尋或混合搜尋來檢索。理賠政策也是如此；但對於病患的理賠紀錄（這些紀錄格式良好且已儲存於 MongoDB 中），我們可能只需要在將其傳遞給 LLM 之前，移除個人可識別資訊（PII）即可。接著，我們要思考什麼檢索技術最適合每個資料來源。你可能在資料處理步驟就已經開始思考這個問題了，但讓我們來快速將其正式化。[00:10:42 → 00:11:33]

臨床指南和理賠政策非常適合向量搜尋（vector search），但它們可能包含一些醫學術語，例如診斷代碼、程序代碼，這些可能無法透過向量搜尋被找出。不過這些資訊可以透過詮釋資料過濾器（metadata filters）或像是關鍵字搜尋（keyword search）的方式被找出來。因此，針對這些資料，你會想使用帶有詮釋資料預先過濾功能的向量搜尋，或是混合搜尋。至於取得病患的理賠紀錄，你可以針對病患的姓名或 ID（或是系統中的任何病患識別碼）進行精確比對（exact match）。接下來，你要思考系統架構會長什麼樣子。[00:11:33 → 00:12:18]

你可能很想直接著手打造一個代理人（agent）。現在對代理人有著大量的炒作，或是你可能想讓程式設計代理人來決定系統架構該長什麼樣；但這樣做，你有最終打造出一個過度設計（over-engineered）系統的風險。所以你應該做的，是從最簡單的設計開始，對其進行評估，在評估過程中發現不足之處，然後從那裡繼續迭代。為了決定系統該長什麼樣子，我們首先來描繪一份保險理賠在系統中的流動過程，因為這將有助於我們了解架構。所以，系統首先會收到理賠請求以及醫師提供的任何臨床紀錄。[00:12:16 → 00:13:03]

接著，系統需要檢索與當前理賠相關的任何臨床指南和理賠政策，並擷取病患的理賠紀錄。然後，所有這些資訊會與理賠文件本身，以及指示 LLM 該如何產生建議的系統提示（system prompts），一起被傳遞給 LLM。LLM 會使用這些資訊，針對理賠是否應該被核准或拒絕產生一項建議。現在，如果這是一個複雜的案例，我們的系統或此時的 LLM 應該將其初步建議傳送給資深醫師。如果 LLM 的建議是拒絕理賠，那麼這也應該升級給人類醫療審查員進行處理。最後，最終決策應連同理賠核准或拒絕的理由一起被記錄到 MongoDB（這也是我們儲存理賠文件的地方）。[00:12:59 → 00:14:06]

現在，我們來談談當今 AI 系統中最常見的一些 AI 設計模式，看看哪些模式可能適用於我們的系統。第一個是 RAG，這是檢索增強生成（Retrieval-Augmented Generation）的縮寫。你們現在可能都知道了。基本上，在這些系統中，你就是利用外部知識來源的資訊來增強 LLM 事先訓練好的知識。[00:14:10 → 00:14:37]

然後是 AI 代理人，你賦予單一 LLM 或多個 LLM（在多代理人系統的情況下）完全的自主權，藉助一組工具來完成端到端（end-to-end）的任務。再來是代理式系統（agentic systems），這涵蓋了廣泛的半自主 AI 系統。這類系統常見的架構模式是控制流（control flows），其中 LLM 在工作流程內執行特定任務，但工作流程中的步驟順序本身是由設計或程式碼預先決定的。接著是將 LLM 作為路由器（router），LLM 的自主權有限，其工作僅是將傳入的請求進行分類，並將它們路由到不同的下游工作流程中。然後還有「人類在迴路中」（human in the loop），LLM 或代理人可以採取某些動作，但在此過程中的某處需要人類的介入。[00:14:38 → 00:15:45]

最後是微調（fine-tuning）。如果你觀察到 LLM 的失敗主要是行為上的，而不是資料或編排（orchestration）上的問題，或是你需要在一項領域特定的任務上獲得卓越效能時，微調會是一個值得探索的好技術。我沒有時間在這裡深入探討這些，但如果你想了解更多關於這些設計模式的資訊，我會把這份資源留置於此供你日後參考。[00:15:42 → 00:16:13]

所以，我們已經思考了這個系統的工作流程應該是什麼樣子。我們也討論了幾種不同的 AI 設計模式。現在，讓我們將其應用並決定我們的系統需要實作哪些設計模式。因為我們需要檢索臨床指南、理賠政策等資料來作為決策的參考，這看起來絕對像是一個 RAG 元件。因此我們需要某種形式的 RAG。[00:16:18 → 00:16:46]

工作流程是預先結構化的，LLM 首先針對理賠是否應獲批准提出建議，並且在非常具體且定義明確的情況下，它需要將理賠案件交給人類審查員處理。這聽起來像是一個受控工作流程（control flow）。你也可以說這看起來有點像將 LLM 當作路由器，特別是如果你需要額外呼叫 LLM 來判定什麼構成複雜案例（除非這已經有明確定義）。所以，是的，這可能是一個作為路由器的 LLM，但現在我們先將其視為控制流。最後，其中還包含「人類在迴路中」的元素，對吧？[00:16:44 → 00:17:26]

因為人類審查員本身就是工作流程中不可或缺的一部分。最後，在設計階段，你也會想思考終端使用者的體驗，以及你將如何從使用者那裡收集回饋以改進系統。在現階段，以下是一些你可以問自己的問題，幫助你思考使用者體驗（UX）和回饋機制該長什麼樣子。首先，系統接收什麼作為輸入？在我們的案例中，某種理賠申請表將會是輸入。[00:17:26 → 00:17:58]

那麼，系統會產生什麼樣的輸出？在我們的案例中，它會是一個核准或拒絕的裁定結果，並附上理賠被核准或拒絕的原因。系統設置在哪裡？它有獨立的應用程式嗎？它是嵌入在現有網站中嗎？[00:17:56 → 00:18:15]

還是它是一個 Slack 機器人？諸如此類。針對我們的應用程式，它很有可能會嵌入在 MDB Health 的主網站中。什麼會觸發系統？在我們的案例中非常明確，是提交理賠申請。[00:18:15 → 00:18:35]

人類的角色是什麼？在我們的案例中，他們將審查系統的評估，並在某些情況下做出最終決定。系統如何解釋自己的決策？在我們的案例中，它是透過提供引用（citations）來解釋，也就是引用為系統評估提供依據的臨床指南和理賠政策。最後，它如何接收回饋？[00:18:33 → 00:19:02]
